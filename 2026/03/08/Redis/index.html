Redis

一.概念

redis本质上是个远程字典服务，所有读写命令等都在一个线程上完成。通常用做用户和mysql的中间层。

例如：当用户查询达到1s 一万次查询，但是mysql只能提供1s 五千次查询，这时候mysql根本顶不住这么大的查询量，因此需要一个中间缓存抵挡大部分的查询量 （没有什么是加一层不能解决的，如果有，那就再加一层）

多种实例需要的是同一份远程缓存，因此不必要重复加内存条，只用一种服务即可，为了避免并发过程中的问题，将这些实例统一塞到同一个线程上。

redis内存入的数据越来越多，迟早有一天会封顶，因此有一种内存过期策略，淘汰过期的数据；当然，也可以用**缓存淘汰（LRU）**将最近最少使用的内存删掉，解决了内存过大的问题

但是，缓存服务重启，内存就全部丢失了，这时候，redis的持久化保证十分重要。RDB，是通过在缓存服务中加入异步线程，定期将全量内存数据定期持久化到磁盘文件里，将内存数据生成快照保存到文件。AOF，是每次写入数据的时候，顺手将数据记录到文件缓存中，并每秒将文件缓存刷入磁盘。

当然，redis拥有redis-cli（命令行工具）可以让用户输入一些指令，读写redis里各种内存数据

Redis事务

1.redis中不会因为某个命令执行失败而导致其他命令无法执行

2.redis在执行命令前，这些命令会被放在一个队列中缓存起来不会立刻被执行

3.事务执行中，其他客户端提交的命令请求，不会被被插入到事务的命令序列中

Redis持久化

1.RDB RDB 是一种按时间间隔将内存中的数据生成快照，并写入磁盘的持久化方式

  • 在指定的时间间隔内（例如每 5 分钟），或者当执行指定数量的写命令后，Redis 会 fork 出一个子进程。
  • 子进程将内存中的数据全量写入到一个临时的 RDB 文件中。
  • 写入完成后，替换旧的 RDB 文件

2.AOF AOF 是一种记录每次写操作命令，并在 Redis 启动时重新执行这些命令来恢复数据的持久化方式

  • 每当 Redis 执行一个改变数据的命令（如 SET、DEL），它会将该命令以 Redis 协议格式追加到 AOF 文件的末尾。
  • 根据配置的同步策略（appendfsync），Redis 会将缓冲区的内容同步到磁盘。
    • always：每次写命令都同步，最安全，但速度慢。
    • everysec（默认）：每秒同步一次，兼顾性能与安全（可能会丢失 1 秒的数据）。
    • no：由操作系统决定何时同步，速度快但风险高。
  • 随着时间推移，AOF 文件会越来越大。Redis 提供了 AOF 重写（rewrite） 机制，可以压缩 AOF 文件

哨兵模式
